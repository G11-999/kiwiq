# MongoDB Path-Based Document Management

## 1. Introduction to AsyncMongoDBClient

AsyncMongoDBClient is a modern, asynchronous interface for MongoDB designed for structured document management using a hierarchical path-based approach. The client handles MongoDB operations in a fully asynchronous way, making it ideal for high-performance applications built with asyncio.

```python
client = AsyncMongoDBClient(
    uri="mongodb://username:password@localhost:27017",
    database="my_app_db",
    collection="structured_documents",
    segment_names=["org", "project", "namespace", "object"],
    text_search_fields=["title", "description", "tags"]
)
```

### Design Philosophy

The client follows these key principles:

- **Path-based organization**: Documents are organized in hierarchical paths for intuitive data structure
- **Strong security**: Prevention of NoSQL injection attacks through careful query construction
- **Flexible permissions**: Granular access control with wildcard pattern support
- **Full asyncio support**: All operations are non-blocking for high-performance applications
- **Batch operations**: Optimized for handling multiple documents efficiently

## 2. Key MongoDB Entities Explained

### Collections
- Similar to tables in relational databases
- Group related documents together
- Typically one collection per entity type
- Our client works with a single collection specified at initialization

### Documents
- Schema-flexible JSON-like records (BSON format)
- Each document has a unique `_id` field (ObjectId by default)
- In our client, documents are organized using path segments as fields
- Contains both metadata (path segments) and actual payload data

### Indexes
- Improve query performance for frequently used fields
- Support for compound, text, and other specialized indexes
- The client automatically creates indexes on segment fields and text search fields

### Queries
- MongoDB uses a rich query language based on JSON
- Supports complex filtering, projection, and aggregation
- The client abstracts query building for path-based operations

## 3. Path-Based Document Model

The core concept of this client is organizing documents in hierarchical paths:

```
org/project/namespace/object
```

### Path Segments

- Each segment in the path becomes a field in the document
- Path segments are defined during client initialization:
  ```python
  segment_names=["org", "project", "namespace", "object"]
  ```
- Documents are stored with this structure:
  ```json
  {
    "_id": ObjectId("..."),
    "org": "acme",
    "project": "website",
    "namespace": "config",
    "object": "settings",
    "data": { "theme": "dark", "features": ["comments", "search"] }
  }
  ```

### Path ID Generation

The client generates document IDs based on the path segments, which ensures:

1. Document uniqueness at each path
2. Efficient lookups without needing to query by individual segments
3. Consistent ID generation for the same path

```python
# Paths generate consistent IDs
path = ["acme", "website", "config", "settings"]
doc_id = client._path_to_id(path)  # Internal method, not typically called directly

# Same path always generates the same ID
same_path = ["acme", "website", "config", "settings"]
same_id = client._path_to_id(same_path)  # Will equal doc_id
```

The ID is generated by joining path segments with a secure delimiter that's unlikely to appear in segment names. The client validates segments to ensure they don't contain the delimiter.

### Variable Path Lengths

The client supports paths of different lengths, from a single segment to the maximum defined by `segment_names`:

```python
# Single segment path
await client.create_object(
    path=["rootdoc"],
    data={"key": "value"}
)

# Two segment path
await client.create_object(
    path=["org", "settings"],
    data={"global_config": true}
)

# Full length path
await client.create_object(
    path=["acme", "website", "config", "settings"],
    data={"theme": "dark"}
)
```

### Pattern Matching

Paths support wildcard patterns for flexible querying:

```python
# List all settings objects across all projects
await client.list_objects(pattern=["acme", "*", "config", "settings"])

# Get all objects in the testing namespace
await client.list_objects(pattern=["*", "*", "testing", "*"])
```

The client also supports embedded wildcards within segments:

```python
# Match objects with names starting with "user_"
await client.list_objects(pattern=["acme", "users", "profiles", "user_*"])
```

```python
async def process_data():
    client = AsyncMongoDBClient(...)
    try:
        await client.setup()  # Create necessary indexes
        # Perform operations...
    finally:
        await client.close()  # Always close connections
```

### Connection Resilience

The client includes automatic retry logic for handling transient connection issues:

```python

try:
    result = await client.fetch_object(path=["acme", "project", "settings"])
    # If the first attempt fails due to a connection issue, the client
    # will automatically reconnect and retry the operation once

    # Only raised if retry also fails
    logging.error(f"Operation failed after retry: {e}")
```

### Batch Operations Performance

Batch operations provide significant performance benefits for handling multiple documents:

```python
# Test performance comparison between batch and individual operations
import time

# Using batch operation
batch_size = 50
paths_data = [
    ([f"org", f"batch{i}", "test", "obj"], {"data": i})
    for i in range(batch_size)
]

start_time = time.time()
await client.batch_create_objects(paths_data)

start_time = time.time()

    await client.create_object(path, data)

print(f"Individual operations time: {individual_time:.3f}s")
print(f"Performance ratio: {individual_time / batch_time:.2f}x")
```

Batch operations typically perform 5-10x faster than individual operations for large sets of documents.

## 8. Security Features

### NoSQL Injection Prevention

MongoDB is vulnerable to NoSQL injection attacks if user input is directly incorporated into database queries. The AsyncMongoDBClient implements several layers of protection against such attacks:

1. **Input Validation**: All path segments and patterns are validated before use.
   ```python
   # Path segments are validated to ensure they're valid strings
   # and don't contain the path delimiter or invalid characters
   try:
       await client.create_object(
           path=["org", "project; db.dropDatabase()", "namespace", "object"],
           data={"test": "data"}
       )
   except ValueError:
       print("Invalid path segment detected and rejected")
   ```

2. **Query Building**: Queries are built using field:value pairs rather than dynamic string concatenation.
   ```python
   # Internally, the client constructs queries like:
   # {"org": "acme", "project": "website"}
   # NOT string concatenation like:
   # "{"org": "" + org_value + ""}" which would be vulnerable
   ```

3. **Parameter Binding**: MongoDB driver parameter binding is used for all queries.
   ```python
   # The client never evaluates expressions or constructs JavaScript
   # All operations use proper parameter binding
   ```

4. **Path Delimiter Security**: The path delimiter is carefully chosen and validated.
   ```python
   # The client rejects path segments containing the delimiter
   try:
       await client.create_object(
           path=["org", "project___with___delimiter", "namespace", "object"],
           data={"test": "data"}
       )
   except ValueError:
       print("Path segment containing delimiter rejected")
   ```

### Permission Model Security

The permission model provides granular access control:

```python
# Permission prefixes define allowed path patterns
allowed_prefixes = [
    ["org1", "project1"],         # Allow org1/project1/*
    ["org1", "project2", "docs"],  # Allow org1/project2/docs/*
    ["org2", "*", "public"]       # Allow org2/*/public/*
]

# Critical security note: Wildcards can only match segments, not ignore them
# ["org1", "*"] matches "org1/any-value" but NOT "org1"
# ["*"] is a special case that grants access to everything
```

Security considerations for permissions:

1. The permission model is implemented on both client and server sides:
   - Client-side check for immediate rejection of clearly unauthorized requests
   - Server-side enforcement via query filters to ensure security at the database level

2. Pattern security follows the principle of least privilege:
   ```python
   # More specific prefixes are safer than wildcards
   # Safer:
   allowed_prefixes = [["org", "project", "namespace"]]
   # Less safe:
   allowed_prefixes = [["org", "*", "*"]]
   ```

3. Permission validation is strict with no fallbacks:
   ```python
   # If permission is denied, operations fail securely
   try:
       result = await client.create_object(
           path=["org1", "project3", "config", "settings"],
           allowed_prefixes=[["org1", "project1"], ["org1", "project2"]]
       )
   except ValueError as e:
       print(f"Permission denied: {e}")
   ```

### Dynamic Permission Management
```python
# User-specific permissions based on role
async def get_user_documents(user_id, user_role):
    allowed_prefixes = []
    
    if user_role == "admin":
        allowed_prefixes = [["org", "*"]]  # Admin sees everything
    elif user_role == "project_manager":
        allowed_prefixes = [[f"org", user_id, "*"]]  # Manager sees their projects

        pattern=["org", "*", "*", "*"],
        allowed_prefixes=allowed_prefixes
    )
```

### Handling Different Data Types

```python
async def store_data(key, value):

    if isinstance(value, dict):

    elif isinstance(value, str):
        path = base_path + ["strings", key]

    else:

    return path
```

## 12. Batch Operations

### Efficient Bulk Processing
The client supports batch operations for improved performance when working with multiple documents:

```python
# Create multiple documents in a single operation
path_data_pairs = [
    (["acme", "website", "config", "db"], {"host": "db1.example.com", "port": 5432}),
    (["acme", "website", "config", "cache"], {"host": "cache.example.com", "port": 6379}),
    (["acme", "website", "config", "api"], {"key": "abc123", "rate_limit": 1000})
]
object_ids = await client.batch_create_objects(path_data_pairs)

# Update multiple documents
update_pairs = [
    (["acme", "website", "config", "db"], {"host": "db2.example.com", "port": 5432}),
    (["acme", "website", "config", "api"], {"key": "xyz789", "rate_limit": 2000})
]
updated_ids = await client.batch_update_objects(update_pairs)

# Create or update multiple documents
upsert_pairs = [
    (["acme", "website", "config", "db"], {"host": "db3.example.com", "port": 5432}),
    (["acme", "website", "config", "newservice"], {"host": "newservice.example.com"})
]
results = await client.batch_create_or_update_objects(upsert_pairs)
for obj_id, created in results:
    print(f"Object {obj_id}: {'Created' if created else 'Updated'}")

# Fetch multiple documents by path
paths = [
    ["acme", "website", "config", "db"],
    ["acme", "website", "config", "cache"],
    ["acme", "website", "config", "api"]
]
documents = await client.batch_fetch_objects(paths)
for path, doc in documents.items():
    if doc:
        print(f"Found document at {path}: {doc['data']}")
    else:
        print(f"No document found at {path}")

# Delete multiple documents by path
delete_paths = [
    ["acme", "website", "config", "db"],
    ["acme", "website", "config", "api"]
]
delete_results = await client.batch_delete_objects(delete_paths)
for path, deleted in delete_results.items():
    print(f"{path}: {'Deleted' if deleted else 'Not found'}")
```

### When to Use Batch Operations

Batch operations significantly improve performance by reducing network overhead and database load. Consider using them when:

1. **Creating or importing many documents at once**: Initial data loading, migrations, or bulk imports
2. **Synchronizing data from another system**: Keeping multiple systems in sync efficiently
3. **Bulk updates**: Applying the same change to many documents or updating related documents together
4. **High-throughput applications**: When performance is critical and operations can be grouped

### Performance Benefits

- **Reduced Network Round-trips**: A single network request instead of many individual ones
- **Better Database Utilization**: MongoDB processes batches more efficiently than individual operations
- **Connection Pooling Optimization**: Fewer connections needed for the same amount of work
- **Automatic Chunking**: Large batches are automatically processed in chunks to avoid overwhelming MongoDB

### Batch Operation with Error Handling


## 13. Edge Cases and Limitations

### Path Length Limitations

The client relies on predefined segment names, which limits the maximum path depth:

```python
# If segment_names = ["org", "project", "namespace", "object"]
# This works:
await client.create_object(
    path=["acme", "website", "config", "settings"],
    data={"theme": "dark"}
)

# This fails (too many segments):
try:
    await client.create_object(
        path=["acme", "website", "config", "settings", "extra"],
        data={"theme": "dark"}
    )
except ValueError as e:
    print(f"Path too long: {e}")
```

### MongoDB Document Size Limit

MongoDB has a 16MB document size limit which applies to objects stored with this client:

```python
# For very large data, consider splitting into multiple documents
# or using GridFS (not directly supported by this client)
try:
    very_large_data = {"content": "x" * 20000000}  # ~20MB
    await client.create_object(
        path=["org", "large_data", "test", "too_big"],
        data=very_large_data
    )
except Exception as e:
    print(f"Document too large: {e}")
```

### Index Limitations

Text search requires proper index setup:

```python
# Text search will fail if setup() hasn't been called
# or if text_search_fields weren't specified at initialization
try:
    results = await client.search_objects(
        key_pattern=["org", "*", "*", "*"],
        text_search_query="important information"
    )
except ValueError as e:
    print(f"Text search failed: {e}")
```

### Handling Special Characters

Some characters require special handling in MongoDB queries:

```python
# Documents with dots in field names
# (MongoDB doesn't allow dots in field names by default)
try:
    await client.create_object(
        path=["org", "project", "config", "field.with.dots"],
        data={"value": 123}
    )
except Exception as e:
    print(f"Error with dots in path: {e}")

# Using $ and other special characters
# (These have special meaning in MongoDB queries)
try:
    await client.create_object(
        path=["org", "project", "config", "$settings"],
        data={"value": 123}
    )
except Exception as e:
    print(f"Error with special character in path: {e}")
```

The client balances security, performance, and flexibility while providing an intuitive interface for asyncio-based applications.

Key strengths include:

1. **Hierarchical Organization**: Intuitive path-based structure for document management
2. **Security**: Built-in protection against NoSQL injection and granular permission controls
3. **Performance**: Optimized batch operations and efficient query construction
4. **Flexibility**: Support for various data types and complex query patterns
5. **Resilience**: Automatic retry for transient connection issues

The client is ideal for applications that need structured document storage with hierarchical organization, such as:

- Configuration management systems
- Content management platforms
- Multi-tenant applications with segregated data
- Document-based workflows with permission controls
- Hierarchical data with complex access patterns
