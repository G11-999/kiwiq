# Usage Guide: ToolExecutorNode

This guide explains how to configure and use the `ToolExecutorNode` to execute tool calls from Large Language Models (LLMs) and other workflow nodes in your workflows.

## Purpose

The `ToolExecutorNode` allows you to:
- Execute tool calls generated by LLM nodes that have tool calling enabled
- Run multiple tools concurrently for better performance
- Control execution limits, timeouts, and error handling
- Map additional input data from your workflow to tool inputs
- Track detailed execution metadata including success/failure rates and timing
- Manage tool access and security through configuration restrictions

## What Are Tools?

In the workflow system, "tools" are special workflow nodes that can be called by AI models or other nodes to perform specific tasks. Tools are like functions that the AI can use to interact with your data and systems.

### Types of Tools Available

1. **Document Management Tools**:
   - `edit_document`: Modify document content with JSON/text operations
   - `view_documents`: Read document content and metadata
   - `search_documents`: Search within documents using AI-powered search
   - `list_documents`: Browse and discover documents

2. **Custom Business Tools**:
   - Any workflow node marked as `node_is_tool: true`
   - Tools you build for your specific business needs

3. **Provider-Inbuilt Tools** (handled by LLM nodes):
   - Web search capabilities
   - Code execution environments
   - Other AI provider-specific tools

## Configuration (`NodeConfig`)

The `ToolExecutorNode` offers extensive configuration to control how tools are executed.

```json
{
  "nodes": {
    "execute_tools": {
      "node_id": "execute_tools",
      "node_name": "tool_executor",      // ** Must be "tool_executor" **
      "node_config": {
        // --- Execution Control ---
        "default_timeout": 30.0,          // Seconds to wait for each tool (default: 30.0)
        "max_concurrent_executions": 5,   // How many tools to run at once (default: 5)
        "continue_on_error": true,         // Keep running other tools if one fails (default: true)
        "include_error_details": true,     // Include full error info in output (default: true)
        
        // --- Tool Call Limits ---
        "tool_call_limit": null,           // Max tool calls allowed (null = no limit)
        "consider_failed_calls_in_limit": true, // Count failed calls toward limit (default: true)
        
        // --- Tool Configuration ---
        "tool_configs": [                  // Optional: Configure specific tools
          {
            "tool_name": "edit_document",
            "version": "2.0.0",           // Use specific version (null = latest)
            "tool_config": {               // Pass config to the tool
              "max_document_size_mb": 5.0
            },
            "map_executor_input_fields_to_tool_input": true, // Override global mapping
            "mappings": {                  // Custom field mappings
              "user_id": "entity_username",
              "context": "view_context"
            }
          }
        ],
        "restrict_tools_to_configured_tools_only": false, // Only allow configured tools
        
        // --- Field Mapping ---
        "map_executor_input_fields_to_tool_input": true // Global field mapping setting
      }
    }
  }
}
```

### Key Configuration Sections:

1. **Execution Control**:
   - `default_timeout`: How long to wait for each tool to complete (30 seconds default)
   - `max_concurrent_executions`: Run multiple tools at the same time for speed (5 default)
   - `continue_on_error`: Don't stop if one tool fails - keep running others (true default)
   - `include_error_details`: Show detailed error messages for debugging (true default)

2. **Tool Call Limits** (Cost & Safety Control):
   - `tool_call_limit`: Maximum number of tool calls allowed per execution
   - `consider_failed_calls_in_limit`: Whether failed attempts count toward the limit
   - Useful for preventing runaway AI tool usage that could be expensive

3. **Tool Configuration**:
   - `tool_configs`: Array of specific tool configurations
   - Each tool can have its own version, configuration, and input mapping rules
   - `restrict_tools_to_configured_tools_only`: Security feature to only allow pre-approved tools

4. **Field Mapping**:
   - `map_executor_input_fields_to_tool_input`: Automatically pass workflow data to tools
   - Custom `mappings` to rename fields between your workflow and tools
   - Powerful feature for connecting workflow context to tool inputs

## Input (`ToolExecutorNodeInputSchema`)

The Tool Executor receives tool calls and additional context:

- **`tool_calls`** (List[ToolCall], Required): Tool calls to execute, usually from an LLM node
- **`execution_timeout`** (float, Optional): Override the default timeout for this execution (default: None)
- **`prior_successful_calls`** (int, Optional): Count of successful calls from previous executions (default: 0)  
- **`prior_failed_calls`** (int, Optional): Count of failed calls from previous executions (default: 0)
- **Additional Fields**: Any extra fields you provide will be mapped to tool inputs if field mapping is enabled

**Critical for Document Tools:**
- **`entity_username`** (str): Required by document tools, automatically mapped from workflow state
- **`view_context`** (Dict): Document serial number mappings, automatically mapped from workflow state

### Tool Call Format

Each `ToolCall` contains:
```json
{
  "tool_name": "edit_document",        // Name of the tool to execute
  "tool_input": {                      // Input data for the tool
    "document_identifier": {
      "doc_key": "brief",
      "docname": "my_brief_123"
    },
    "operations": [...]
  },
  "tool_id": "call_abc123"            // Unique identifier (optional)
}
```

## Output (`ToolExecutorNodeOutputSchema`)

The Tool Executor provides comprehensive results:

- **`tool_outputs`** (List[ToolOutput]): Results from each tool execution
- **`tool_call_metadata`** (List[ToolCallMetadata]): Detailed execution information
- **`total_execution_time`** (float): Total time for all tool executions
- **`successful_calls`** (int): Number of tools that completed successfully  
- **`failed_calls`** (int): Number of tools that failed
- **`state_changes`** (Dict, Optional): Any state modifications made by tools

### Tool Output Format

Each `ToolOutput` contains:
```json
{
  "tool_call_id": "call_abc123",       // Matches the tool call ID
  "content": "Operation completed...", // Result content from the tool
  "name": "edit_document",             // Tool that was executed
  "type": "tool",                      // Always "tool"
  "status": "success",                 // "success" or "error"
  "error_message": null,               // Error details if failed
  "state_changes": {                   // Document serial numbers for next iteration
    "content_pillars_doc_1": {
      "docname": "content_pillars_doc",
      "version": "default"
    }
  }
}
```

### Document Serial Numbers and View Context

Document tools use a serial number system for referencing documents across workflow iterations:

**Initial State:**
```json
{
  "view_context": {}  // Empty at start
}
```

**After viewing documents:**
```json
{
  "view_context": {
    "content_pillars_doc_1": {
      "docname": "content_pillars_doc",
      "version": "default"
    },
    "user_dna_doc_2": {
      "docname": "user_dna_doc", 
      "version": "default"
    }
  }
}
```

**AI uses serial numbers in subsequent calls:**
```json
{
  "tool_name": "edit_document",
  "tool_input": {
    "document_identifier": {
      "doc_key": "content_pillars_doc",
      "document_serial_number": "content_pillars_doc_1"  // References previous view
    },
    "operations": [...]
  }
}
```

## Real-World Example: Natural Language Document Editing Agent

The following example shows a complete AI agent workflow that uses the Tool Executor for document management. This is based on a production workflow for natural language document editing with human-in-the-loop approval.

### Complete Workflow Configuration

```json
{
  "nodes": {
    // LLM Node with Document Tools
    "llm_with_tools": {
      "node_id": "llm_with_tools",
      "node_name": "llm",
      "node_config": {
        "llm_config": {
          "model_spec": {
            "provider": "anthropic",
            "model": "claude-sonnet-4-20250514"
          },
          "temperature": 0.7,
          "max_tokens": 2000
        },
        "tool_calling_config": {
          "enable_tool_calling": true,
          "parallel_tool_calls": true
        },
        "tools": [
          {
            "tool_name": "edit_document",
            "is_provider_inbuilt_tool": false,
            "provider_inbuilt_user_config": {}
          },
          {
            "tool_name": "view_documents", 
            "is_provider_inbuilt_tool": false,
            "provider_inbuilt_user_config": {}
          },
          {
            "tool_name": "search_documents",
            "is_provider_inbuilt_tool": false,
            "provider_inbuilt_user_config": {}
          },
          {
            "tool_name": "list_documents",
            "is_provider_inbuilt_tool": false,
            "provider_inbuilt_user_config": {}
          }
        ],
        "output_schema": {
          "schema_definition": {
            // Structured output for workflow control
            "type": "object",
            "properties": {
              "action": {
                "type": "string",
                "enum": ["continue", "end_workflow", "ask_clarification"]
              },
              "reasoning": {
                "type": "string",
                "description": "Explanation of the chosen action"
              }
            }
          }
        }
      }
    },
    
    // Tool Executor with Production Configuration
    "tool_executor": {
      "node_id": "tool_executor",
      "node_name": "tool_executor",
      "node_config": {
        "default_timeout": 30.0,
        "max_concurrent_executions": 3,
        "continue_on_error": true,
        "include_error_details": true,
        "map_executor_input_fields_to_tool_input": true
      }
    },
    
    // Human-in-the-Loop Approval Node
    "hitl_approval": {
      "node_id": "hitl_approval",
      "node_name": "hitl_node__default",
      "node_config": {},
      "dynamic_output_schema": {
        "fields": {
          "user_action": {
            "type": "enum",
            "enum_values": ["approve_tools", "deny_tools", "provide_clarification", "stop_workflow"],
            "required": true,
            "description": "User's decision on tool execution or clarification"
          },
          "user_feedback": {
            "type": "str",
            "required": false,
            "description": "Additional feedback or clarification from user"
          }
        }
      }
    }
  },
  
  "edges": [
    // LLM generates tool calls
    {
      "src_node_id": "llm_with_tools",
      "dst_node_id": "tool_executor",
      "mappings": [
        {"src_field": "tool_calls", "dst_field": "tool_calls"}
      ]
    },
    
    // State provides context to tool executor (CRITICAL for document tools)
    {
      "src_node_id": "$graph_state",
      "dst_node_id": "tool_executor",
      "mappings": [
        {"src_field": "latest_tool_calls", "dst_field": "tool_calls"},
        {"src_field": "entity_username", "dst_field": "entity_username"},
        {"src_field": "view_context", "dst_field": "view_context"}
      ]
    },
    
    // Tool executor updates state with results
    {
      "src_node_id": "tool_executor",
      "dst_node_id": "$graph_state",
      "mappings": [
        {"src_field": "tool_outputs", "dst_field": "latest_tool_outputs"},
        {"src_field": "state_changes", "dst_field": "view_context"}
      ]
    }
  ]
}
```

## Best Practices

### 1. Tool Call Limits
```json
{
  "tool_call_limit": 10,                    // Prevent runaway tool usage
  "consider_failed_calls_in_limit": true    // Count failures to avoid retry loops
}
```

**Why**: AI models can sometimes make excessive tool calls, leading to high costs and slow performance. Setting limits protects your system.

### 2. Error Handling
```json
{
  "continue_on_error": true,        // Don't stop on first failure
  "include_error_details": true     // Get full error info for debugging
}
```

**Why**: Tools can fail for various reasons (network issues, validation errors, etc.). Continuing execution allows other tools to succeed.

### 3. Field Mapping for Context
```json
{
  "map_executor_input_fields_to_tool_input": true,
  "tool_configs": [
    {
      "tool_name": "edit_document",
      "mappings": {
        "current_user": "entity_username",    // Map workflow user to tool field
        "session_data": "view_context"        // Pass session context to tools
      }
    }
  ]
}
```

**Why**: Tools often need context from your workflow (user info, session data, etc.). Field mapping automatically provides this context.

### 4. Security with Tool Restrictions
```json
{
  "restrict_tools_to_configured_tools_only": true,
  "tool_configs": [
    // Only list tools you want to allow
    {"tool_name": "view_documents"},
    {"tool_name": "search_documents"}
    // edit_document and list_documents are blocked
  ]
}
```

**Why**: In production environments, you may want to limit which tools AI can access for security and compliance reasons.

### 5. Performance Optimization
```json
{
  "max_concurrent_executions": 10,   // Higher for I/O bound tools
  "default_timeout": 60.0            // Longer for complex operations
}
```

**Why**: Document operations can be I/O intensive. Higher concurrency speeds up multiple tool calls, while appropriate timeouts prevent hanging.

## Troubleshooting

### Tool Not Found Errors
**Problem**: "Tool 'my_tool' not found in registry"
- **Solution**: Ensure the tool is registered and the name matches exactly
- **Check**: Tool node has `node_is_tool: true` set

### Timeout Issues
**Problem**: Tools failing with timeout errors
- **Solution**: Increase `default_timeout` or set `execution_timeout` in input
- **Consider**: Breaking large operations into smaller tool calls

### Field Mapping Problems
**Problem**: Tools receiving wrong or missing input data
- **Solution**: Check field names match between workflow and tool schemas
- **Use**: Custom `mappings` to rename fields as needed

### Tool Call Limit Reached
**Problem**: "Tool call limit reached" errors
- **Solution**: Increase `tool_call_limit` or optimize AI prompts to use fewer tools
- **Consider**: Whether failed calls should count toward limit

### Permission Errors
